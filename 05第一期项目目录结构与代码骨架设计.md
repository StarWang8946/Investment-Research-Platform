# 第一期项目目录结构与代码骨架设计

> 适用阶段：第一期“研究信息中台 MVP”
> 
> 目标：给第一期最小多 Agent 架构提供可直接开工的项目目录、模块边界和代码骨架设计

---

## 1. 设计目标

第一期虽然是单人、本地开发，但从代码结构上不能把它写成脚本集合。原因很简单：

- 一期已经有 `总控 Agent + 研究 Agent + 报告协作 Agent`
- 一期已经有文档处理、检索、问答、研究资产、导出等多个模块
- 二期和三期还会继续增加因子、回测、风控、组合能力

所以第一期的代码结构必须满足两件事：

- 当前足够轻，能快速开发
- 后续足够稳，不需要推倒重来

---

## 2. 目录设计原则

### 2.1 为什么按模块分层

建议采用“接口层 -> 服务层 -> Agent 层 -> Skill 层 -> 数据访问层”的结构。

原因：

- 接口层负责 HTTP 入参出参，不掺业务逻辑
- 服务层负责业务流程，不直接写 SQL
- Agent 层负责任务理解、路由、结果汇总
- Skill 层负责具体能力实现
- 数据访问层负责和数据库交互

这样拆的好处是：

- 多 Agent 逻辑不会和 API 写在一起
- 检索、摘要、导出等能力可以复用
- 后续替换模型或增加 Agent 成本低

### 2.2 为什么不建议按“功能页面”建目录

不建议一开始按“问答页”“上传页”“日报页”来组织后端代码。

原因：

- 页面会变，能力模块更稳定
- 一份研究备忘录可能同时调用检索、摘要、引用、资产保存
- 如果按页面拆，逻辑会大量重复

## 4. 每层职责说明

## 4.1 `app/main.py`

### 职责

- 创建 FastAPI 应用
- 注册路由
- 初始化日志
- 初始化数据库依赖

### 为什么单独保留极薄入口

- 入口文件越薄，后续维护越稳
- 业务逻辑不应该放在 `main.py`

---

## 4.2 `core/`

### 职责

- 管理配置
- 管理日志
- 管理统一异常
- 管理公共依赖注入

### 推荐文件

- `config.py`
  - 读取环境变量
- `logger.py`
  - 初始化结构化日志
- `exceptions.py`
  - 自定义业务异常
- `deps.py`
  - 数据库 session、当前用户等依赖
- `constants.py`
  - 任务状态、文档类型、资产类型常量

### 为什么先抽 `core`

- 一期后面会频繁增加配置项和异常处理
- 如果这些分散在各模块里，后续会很乱

---

## 4.3 `api/`

### 职责

- 负责 HTTP 层
- 负责参数校验
- 负责调用 service 或 task service
- 不直接操作数据库

### 推荐拆法

- `routers/`
  - 放具体路由
- `schemas/`
  - 放 Pydantic 请求/响应模型

### 为什么 API 层不直接调 Agent

- API 层应该只关心“请求进来怎么转成业务调用”
- Agent 调用属于业务流程，应该由 service 层协调

---

## 4.4 `models/`

### 职责

- 定义 SQLAlchemy ORM 模型
- 对应数据库表

### 为什么 ORM 单独放

- 表结构是稳定资产
- 和业务流程、接口入参不是同一层概念

---

## 4.5 `repositories/`

### 职责

- 封装数据库读写
- 提供文档、任务、资产等对象的查询方法

### 示例职责

- `document_repository.py`
  - 创建文档
  - 更新解析状态
  - 查询文档列表
- `chunk_repository.py`
  - 批量写入 chunks
  - 向量检索
  - 关键词检索

### 为什么必须有 repository 层

- 避免 SQL 到处散落
- 后续如果换查询方式，只改一层

---

## 4.6 `services/`

### 职责

- 承担业务逻辑
- 串联 repository、Agent、Skill、LLM Provider

### 核心服务建议

- `document_service.py`
  - 文档上传与元数据管理
- `ingest_service.py`
  - 解析、切分、embedding、入库
- `retrieval_service.py`
  - 混合检索
- `qa_service.py`
  - 基于检索结果生成回答
- `citation_service.py`
  - 生成并保存引用
- `asset_service.py`
  - 保存研究备忘录和日报
- `export_service.py`
  - 导出 Markdown/DOCX
- `company_service.py`
  - 公司主数据查询
- `tag_service.py`
  - 标签管理和文档/资产标签关系
- `stats_service.py`
  - 文档、检索、资产、任务统计
- `task_service.py`
  - 创建任务、更新任务、记录运行轨迹

### 为什么 service 层是核心

- 一期复杂度主要不在模型，而在流程串联
- Service 层最适合沉淀这些流程

---

## 4.7 `agents/`

### 职责

- 表达多 Agent 架构的任务分工
- 负责任务理解、路由、结果汇总

### 推荐文件

- `base_agent.py`
  - 统一 Agent 接口
- `orchestrator_agent.py`
  - 总控 Agent
- `research_agent.py`
  - 研究 Agent
- `reporting_agent.py`
  - 报告协作 Agent
- `registry.py`
  - Agent 注册表

### 每个 Agent 的职责

#### `OrchestratorAgent`

- 接受任务
- 判断任务类型
- 路由到研究 Agent 或报告协作 Agent
- 汇总结果

#### `ResearchAgent`

- 调用检索、摘要、问答、备忘录 Skill
- 输出研究问答和研究备忘录

#### `ReportingAgent`

- 调用日报模板、导出 Skill
- 输出晨会日报和导出结果

### 为什么要先做 Agent 类而不是只做 service

- 因为当前架构已经明确是多 Agent 方向
- 一期即使只做 3 个 Agent，也要先把“角色边界”立住

---

## 4.8 `skills/`

### 职责

- 封装可复用能力
- 让 Agent 不直接依赖具体实现细节

### 推荐 Skill

- `document_parse_skill.py`
- `chunking_skill.py`
- `embedding_skill.py`
- `keyword_search_skill.py`
- `vector_search_skill.py`
- `hybrid_retrieval_skill.py`
- `rerank_skill.py`
- `summarization_skill.py`
- `citation_trace_skill.py`
- `memo_generation_skill.py`
- `daily_report_skill.py`
- `export_markdown_skill.py`
- `export_docx_skill.py`

### 为什么要单独做 Skill 层

- Skill 是未来扩展量化、风控、组合能力的关键抽象
- 一期把检索、摘要、模板生成抽成 Skill，二期三期才好复用

---

## 4.9 `workflows/`

### 职责

- 表达跨模块标准流程

### 推荐工作流

- `document_ingest_workflow.py`
  - 上传后解析入库
- `qa_workflow.py`
  - 问答流程
- `memo_workflow.py`
  - 备忘录流程
- `daily_report_workflow.py`
  - 日报流程

### 为什么需要 workflow 层

- 工作流不是单个 Skill，也不是单个 Agent
- 把标准流程独立出来，能避免 Service 过于膨胀

---

## 4.10 `llm/`

### 职责

- 统一管理模型调用

### 推荐文件

- `base_provider.py`
  - 定义统一接口
- `provider_factory.py`
  - 根据配置返回 provider
- `external_provider.py`
  - 外部大模型调用
- `embedding_provider.py`
  - embedding 调用

### 为什么单独抽象模型层

- 第一阶段虽然以外部模型为主，但后续可能切换模型
- 不应该把模型 SDK 调用散落在 Skill 或 Agent 中

---

## 4.11 `db/`

### 职责

- 管理数据库连接和迁移

### 推荐文件

- `base.py`
- `session.py`
- `migrations/`

### 为什么要尽早引入迁移目录

- 表结构会随着一期推进调整
- 没有迁移管理，后面会非常痛苦

---

## 4.12 `utils/`

### 职责

- 存放纯工具函数

### 放什么

- 文件哈希
- 时间格式转换
- Markdown 处理
- 文本清洗基础函数

### 不放什么

- 不放数据库逻辑
- 不放业务流程
- 不放 Agent 路由

---

## 5. 推荐核心类设计

## 5.1 `BaseAgent`

### 作用

- 定义所有 Agent 的统一接口

### 建议方法

```python
class BaseAgent:
    name: str

    async def run(self, task_context: dict) -> dict:
        raise NotImplementedError
```

### 为什么这样设计

- 保证所有 Agent 都能被总控 Agent 统一调度

---

## 5.2 `BaseSkill`

### 作用

- 定义所有 Skill 的统一调用方式

### 建议方法

```python
class BaseSkill:
    name: str

    async def execute(self, payload: dict) -> dict:
        raise NotImplementedError
```

### 为什么这样设计

- Agent 调用 Skill 时不需要关心底层实现

---

## 5.3 `AgentRegistry`

### 作用

- 管理可用 Agent

### 建议职责

- 注册 Agent
- 按名称查找 Agent
- 获取默认 Agent

### 为什么需要

- 一期虽然 Agent 不多，但后续会变多
- 有注册表后，总控 Agent 不需要硬编码创建实例

---

## 5.4 `SkillRegistry`

### 作用

- 管理可用 Skill

### 建议职责

- 注册 Skill
- 查询 Skill
- 执行 Skill

### 为什么需要

- 多 Agent 架构下，Skill 是共享能力池
- 先把注册机制立起来，后面好扩展

---

## 5.5 `TaskContext`

### 作用

- 统一任务输入结构

### 建议字段

- `task_id`
- `task_type`
- `input_text`
- `input_payload`
- `current_user`
- `route_agent`

### 为什么要统一上下文对象

- 不然不同 Agent、Service、Workflow 之间传参会越来越乱

---

## 6. 第一期关键调用链

## 6.1 文档上传与入库链路

```text
API POST /api/v1/documents
-> DocumentService.create_document()
-> 写入 documents

API POST /api/v1/documents/{id}/ingest
-> TaskService.create_task(task_type=document_ingest)
-> IngestWorkflow.run()
-> DocumentParseSkill
-> ChunkingSkill
-> EmbeddingSkill
-> ChunkRepository.batch_insert()
-> TaskService.finish_task()
```

### 为什么这样拆

- 文档上传和文档解析是两个不同阶段
- 中间经过 TaskService，后续好切异步
- 接口文档保留上传和入库两个接口，便于本地调试解析失败；前端也可以在上传后立即调用 ingest

---

## 6.2 问答链路

```text
API POST /api/v1/qa/ask
-> TaskService.create_task(task_type=qa)
-> OrchestratorAgent.run()
-> route -> ResearchAgent.run()
-> HybridRetrievalSkill
-> Summarization / QA generation
-> CitationTraceSkill
-> CitationService.save()
-> TaskService.finish_task()
```

### 为什么先走总控 Agent

- 多 Agent 架构的核心就是由总控统一收口
- 后续增加风控、量化、报告类任务时接口不用变
- `/qa/ask` 是同步等待任务完成的便捷接口，底层仍复用任务逻辑并写入 `tasks`

---

## 6.3 研究备忘录链路

```text
API POST /api/v1/qa/memo-generate
-> TaskService.create_task(task_type=memo_generate)
-> OrchestratorAgent
-> ResearchAgent
-> HybridRetrievalSkill
-> MemoGenerationSkill
-> CitationTraceSkill
-> AssetService.create_asset()
```

---

## 6.4 晨会日报链路

```text
API POST /api/v1/qa/daily-report-generate
-> TaskService.create_task(task_type=daily_report)
-> OrchestratorAgent
-> ReportingAgent
-> DailyReportSkill
-> AssetService.create_asset()
-> ExportService.export_markdown/docx()
```

## 8. 第一期最小代码骨架应达到什么程度

### 必须有

- 可运行的 FastAPI 服务
- 完整的数据库连接和模型
- 文档上传和入库流程
- 检索与问答流程
- 总控 Agent 路由
- 研究资产保存
- Prompt 模板管理
- `X-Request-ID` 链路追踪

### 可以后补

- 更细的权限系统
- 更完整的测试覆盖
- 更复杂的工作流状态机
- 批量操作、统计看板和回收站 UI

## 9. 容易写乱的地方

### 9.1 不要把 LLM 调用写到路由里

原因：

- 路由层只应该处理 HTTP

### 9.2 不要把 SQL 写到 Agent 里

原因：

- Agent 应该负责角色逻辑，不负责数据访问细节

### 9.3 不要把检索、摘要、导出混成一个大 Service

原因：

- 一期虽然小，但这些能力未来都会扩展

### 9.4 不要把“研究备忘录”和“日报”都当成普通问答

原因：

- 它们是结构化资产，不只是一次回答
