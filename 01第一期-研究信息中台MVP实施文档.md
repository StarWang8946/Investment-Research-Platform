## 1. 一期范围

### 1.1 必做范围

- 文档上传
- 文档解析
- 文档切分
- 向量化和检索
- 带引用的研究问答
- 研究备忘录生成
- 晨会日报生成
- 研究资产保存

### 1.2 从多 Agent 总架构映射到一期

虽然总架构是多 Agent 模式，但第一期不需要把 1 个总控 + 5 个专业 Agent 全部做完。一期建议只落地 `1 个总控 Agent + 2 个专业 Agent`，其余 Agent 只保留接口，不做完整能力。

#### 一期实际落地的 Agent

- `总控 Agent`
  - 负责意图识别、任务拆解、路由、结果汇总
- `研究 Agent`
  - 负责研报、公告、财报、纪要的检索、摘要、研究备忘录生成
- `报告协作 Agent`
  - 负责日报、周报、纪要、导出和模板化输出

#### 一期只保留接口、不完整实现的 Agent

- `量化 Agent`
  - 先只保留占位，不做因子、回测
- `投资决策 Agent`
  - 先只保留占位，不做组合建议
- `风控 Agent`
  - 先只保留占位，不做风险校验

#### 一期实际落地的 Skill

- 文档解析
- 文档切分
- 向量检索
- 关键词检索
- 混合检索
- 检索重排配置预留
- 摘要生成
- 引用溯源
- 研究备忘录模板
- 晨会日报模板
- 导出 Markdown/DOCX
- Prompt 模板配置

#### 一期暂不落地的 Skill

- 因子计算
- 回测引擎
- 组合优化
- VaR
- 合规校验
- 审批流

### 1.3 明确不做

- 因子计算
- 回测
- 组合优化
- 风控监控
- 交易系统对接

## 2. 一期用户场景

### 场景 1：上传一份研报并提问

```text
上传研报
-> 系统解析文档
-> 切分并入库
-> 生成向量
-> 用户提问
-> 检索相关片段
-> 生成带引用答案
```

### 场景 2：围绕某家公司生成研究备忘录

```text
选择公司
-> 检索历史研报、公告、财报
-> 提取关键信息
-> 生成结构化备忘录
-> 用户手工修订
-> 保存为研究资产
```

### 场景 3：生成晨会日报

```text
选择日期
-> 汇总市场、公告、新闻、已有研究结论
-> 生成日报模板
-> 用户复核
-> 导出 Markdown/DOCX
```

## 4. 一期技术实现

### 4.1 推荐技术栈

|模块|选型|
|---|---|
|前端|Streamlit 或 Next.js|
|后端|FastAPI|
|数据库|PostgreSQL|
|向量检索|pgvector|
|对象存储|本地目录|
|文档解析|pymupdf、python-docx、unstructured|
|模型接入|统一 LLM Adapter|

### 4.2 技术原理

#### RAG

- 文档先切分为片段
- 每个片段生成 embedding
- 问题先检索，再生成答案

作用：

- 降低幻觉
- 让答案依赖本地文档
- 支持引用溯源

#### 混合检索

- 向量检索找语义相近内容
- 关键词检索找术语和精确实体
- 最后重排结果

#### 引用系统

- 每个答案都绑定 `source_id`
- 每个引用都能定位到文档片段
- 最终结果保存引用列表

## 10. 与 API v2 的对齐说明

第一期接口以 [03第一期-接口清单（API 设计）](03第一期-接口清单（API%20设计）.md) 为准，业务接口统一使用 `/api/v1` 前缀。

### 10.1 API 模块

- 系统接口
- 公司主数据接口
- 标签接口
- 文档接口
- 检索与问答接口
- 任务接口
- 研究资产接口
- 模板与配置接口
- 统计看板接口

### 10.2 调用边界

- `/qa/ask` 是同步等待任务完成的便捷入口
- `/tasks` 是统一异步任务入口
- 两类入口都应写入 `tasks` 和 `task_runs`
- `X-Request-ID` 用于请求链路追踪，错误响应中的 `request_id` 与该 Header 对齐

### 10.3 一期落地顺序

- P0：健康检查、系统信息、公司查询、文档上传/查询/chunk、检索、问答、任务创建/查询
- P1：标签、流式问答、备忘录、日报、研究资产、Markdown/DOCX 导出、Prompt 模板
- P2/P3：资产版本、软删除、批量操作、配置接口、统计看板
