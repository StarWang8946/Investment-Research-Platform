# 智能投研平台第一期 — 项目现状评估报告

> 评估日期：2026-06-06
> 评估范围：需求、技术方案、落地可行性
> 评估依据：01-06 号设计文档 + app/ 目录实际代码

## 一、需求评估

### 1.1 需求定义清晰度 ⭐⭐⭐⭐

| 维度 | 评价 |
|------|------|
| MVP 范围 | 明确：文档接入 → 知识检索 → 研究问答 → 备忘录/日报生成 |
| 边界划分 | 清晰：因子计算、回测、组合优化、风控监控、交易系统明确列入第二期 |
| 用户场景 | 具体：上传研报并提问、生成研究备忘录、生成晨会日报，三场景流程描述完整 |
| 验收标准 | 可量化：上传文档后可检索、问答有引用、可生成备忘录/日报、结论可回溯原文 |

### 1.2 需求完整性 ⭐⭐⭐⭐

**覆盖完整的功能域：**

- 文档管理（上传、解析、切分、预览、删除）
- 知识检索（混合检索：关键词 + 向量）
- 研究问答（LLM 驱动 + 引用溯源）
- 研究资产（备忘录、日报、导出）
- 任务管理（研究任务追踪）
- 标签系统（多标签关联）
- 公司主数据（A 股公司基础信息）
- Prompt 模板配置
- 系统配置

**缺失项：**

| 缺失内容 | 影响程度 | 说明 |
|----------|---------|------|
| 前端界面设计 | 中 | 本期为纯后端 API 可接受，但演示需前端 |
| 样本数据准备方案 | 中 | 测试和演示需要 20-50 份真实研报 |
| 用户权限模型 | 低 | 本期为单用户，多用户权限属第二期 |

### 1.3 需求可行性 ⭐⭐⭐⭐⭐

- MVP 范围克制，聚焦「投研信息聚合 + AI 问答」核心价值
- 6 周排期拆解到天，任务粒度合理
- 技术选型务实，无过度设计

## 二、技术方案评估

### 2.1 架构设计 ⭐⭐⭐⭐

文档中的多层架构设计合理：

```
┌─────────────────────────────────────────┐
│              API 层 (FastAPI)            │
├─────────────────────────────────────────┤
│           Service 业务逻辑层             │
├──────────┬──────────┬──────────────────┤
│  Agent   │  Skill   │  Repository 层   │
│  编排层   │  工具层   │  数据访问层      │
├──────────┴──────────┴──────────────────┤
│         LLM 适配层 / DB (PostgreSQL)     │
└─────────────────────────────────────────┘
```

**亮点：**
- 数据库设计优秀：15 张表、82 个索引，覆盖文档/任务/资产/引用/标签/公司主数据
- 多 Agent 架构方向正确：总控 Agent + 研究 Agent + 报告 Agent 协作
- 前瞻性设计：Prompt 模板配置化、系统配置化、X-Request-ID 链路追踪

### 2.2 设计与实现的差距 ⚠️ 关键风险

| 设计文档规划 | 实际代码状态 | 差距等级 |
|------------|------------|---------|
| `models/` ORM 模型层 | ❌ 未实现，直接用 psycopg 写 SQL | 中 |
| `repositories/` 数据访问层 | ❌ 未实现，SQL 散落在 services/ 中 | 中 |
| `agents/` 多 Agent 层 | ❌ 未实现 | 高 |
| `skills/` Skill 层 | ❌ 未实现 | 高 |
| `workflows/` 工作流层 | ❌ 未实现 | 高 |
| `llm/` 模型适配层 | ❌ 未实现 | 高 |
| services/ 服务层 | ⚠️ 部分实现 | 中 |

## 三、落地可行性评估

### 3.1 技术可行性 ⭐⭐⭐⭐

| 维度 | 评估 |
|------|------|
| 技术栈成熟度 | 高：FastAPI / PostgreSQL / pgvector / PyMuPDF 均为成熟方案 |
| 数据库就绪度 | 高：15 张表已建好，可直接继续开发 |
| 外部依赖 | 低：LLM 接入仅需 HTTP 调用，无复杂集成 |
| 部署复杂度 | 低：单机部署即可满足 MVP |

### 3.2 主要风险点

| 风险 | 等级 | 说明 |
|------|------|------|
| LLM 接入工程化 | 🔴 高 | 需处理 API Key 管理、错误重试、超时、Token 限制、流式输出 |
| 向量检索工程化 | 🔴 高 | 需 embedding 流水线、pgvector 索引调优、混合检索策略 |
| 架构偏差 | 🟡 中 | 当前"服务直调 SQL" vs 设计"Agent→Skill→Service→Repository"，需决策是否重构 |
| 引用系统 | 🟡 中 | 编号生成、溯源定位、citations 写入，逻辑不复杂但易遗漏 |
| 无前端 | 🟡 中 | MVP 演示需要可交互界面 |
| 测试不足 | 🟡 中 | 核心接口零覆盖，上线后维护成本高 |

## 五、总结

| 维度 | 评分 | 一句话总结 |
|------|------|----------|
| 需求 | ⭐⭐⭐⭐ | 定义清晰、范围克制、可执行性强 |
| 技术方案 | ⭐⭐⭐⭐ | 架构设计优秀，但实现与设计有差距 |
| 落地可行性 | ⭐⭐⭐⭐ | 技术栈成熟、数据库就绪，核心缺口在 LLM 接入和向量检索 |
| **整体完成度** | **40-50%** | **骨架扎实，核心智能能力（LLM + 向量检索）尚未落地** |

> **关键判断：** 项目「地基打得牢」（数据库、API 骨架、文档解析），但「智能核心未通」（LLM 接入、向量检索、Agent 编排均为空白）。当前最紧迫的是打通 LLM → 向量检索 → 问答 这条主链路，否则项目无法体现核心价值。
