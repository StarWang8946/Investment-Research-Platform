# 第一期数据库表结构设计

> 适用阶段：第一期“研究信息中台 MVP”
>
> 目标：为文档接入、检索问答、研究备忘录、晨会日报、任务路由和引用溯源提供可直接开发的数据库设计
>
> 版本：v2

---

## 1. 设计原则

### 1.1 为什么第一期用关系型数据库为主

第一期建议以 `PostgreSQL` 为主库，原因如下：

- 文档元数据、任务记录、研究资产、引用关系都天然是结构化数据
- 多个实体之间有明确关系，适合用外键和索引约束一致性
- 二期和三期仍然可以继续扩展，不需要迁库
- 配合 `pgvector` 可以在同一数据库中完成向量检索，减少独立组件
- 配合 `pg_trgm` 可以补足标题、关键词、模糊搜索能力

### 1.2 第一期表设计目标

- 支撑文档上传与解析
- 支撑 chunk 切分、结构化抽取与向量化
- 支撑带引用问答
- 支撑总控 Agent 的任务记录
- 支撑研究备忘录与晨会日报沉淀
- 支撑标签检索和公司主数据管理
- 为二期因子与回测扩展预留连接位

### 1.3 第一期不追求的内容

- 不做复杂多租户
- 不做细粒度 ABAC 权限模型
- 不做企业级审计仓
- 不做海量数据分库分表
- 不直接上分区表，但要提前考虑分区策略

## 3. 数据库扩展

### 3.1 必要扩展

- `uuid-ossp` 或应用层生成 UUID
- `pgvector`
- `pg_trgm`

### 3.2 为什么需要 `pgvector`

- 第一阶段就要做 embedding 检索
- 向量直接放在 `document_chunks` 表中，便于统一管理
- 减少外部向量数据库的部署和维护成本

### 3.3 为什么需要 `pg_trgm`

- 投研场景存在大量标题模糊搜、公司简称模糊搜、关键词召回
- 仅靠向量检索不够，模糊匹配是混合检索的重要组成部分

## 5. 详细表结构

## 5.1 `users`

### 用途

- 记录当前系统操作者
- 为任务、研究资产、修订记录提供归属

### 建议字段

|字段|类型|约束|说明|
|---|---|---|---|
|id|uuid|pk|用户主键|
|username|varchar(64)|unique not null|登录名或唯一标识|
|display_name|varchar(128)|not null|展示名|
|role|varchar(32)|not null default 'researcher'|角色|
|status|varchar(16)|not null default 'active'|状态|
|created_at|timestamptz|not null|创建时间|
|updated_at|timestamptz|not null|更新时间|

### 设计原因

- 第一期虽然是单人开发，也要把“人”和“内容”关联起来
- 后续多人协作时不需要重构数据归属模型

---

## 5.2 `company_basic_info`

### 用途

- 维护公司主数据
- 统一 `company_code`、简称、全称、市场和行业口径

### 建议字段

|字段|类型|约束|说明|
|---|---|---|---|
|id|uuid|pk|公司主键|
|company_code|varchar(32)|unique not null|证券代码|
|company_name|varchar(128)|not null|公司全称|
|company_short_name|varchar(128)||公司简称|
|exchange|varchar(32)||交易所|
|market|varchar(32)||市场|
|industry_code_l1|varchar(32)||一级行业编码|
|industry_name_l1|varchar(128)||一级行业名称|
|industry_code_l2|varchar(32)||二级行业编码|
|industry_name_l2|varchar(128)||二级行业名称|
|security_type|varchar(32)||证券类型|
|list_date|date||上市日期|
|delist_date|date||退市日期|
|is_active|boolean|not null default true|是否有效|
|created_at|timestamptz|not null|创建时间|
|updated_at|timestamptz|not null|更新时间|

### 索引建议

- `idx_company_basic_info_short_name`
- `idx_company_basic_info_industry_l1`
- `idx_company_basic_info_is_active`

### 设计原因

- 一期所有地方都在使用 `company_code`
- 没有主数据表，后续公司简称、行业归类和检索都会失控

---

## 5.3 `documents`

### 用途

- 记录上传文档的主信息
- 作为文档解析、切分、检索的入口实体

### 建议字段

|字段|类型|约束|说明|
|---|---|---|---|
|id|uuid|pk|文档主键|
|source_id|varchar(128)|unique not null|文档来源唯一标识|
|title|varchar(512)|not null|文档标题|
|doc_type|varchar(32)|not null|如 `report`、`announcement`、`financial_report`、`memo`|
|source|varchar(128)||来源机构或来源渠道|
|company_id|uuid|fk company_basic_info(id)|公司主数据 ID|
|company_code|varchar(32)||公司代码快照|
|company_name|varchar(128)||公司名称快照|
|industry|varchar(128)||行业快照|
|publish_date|date||发布日期|
|permission_level|varchar(16)|not null default 'internal'|权限级别|
|file_name|varchar(256)|not null|原文件名|
|file_path|text|not null|本地文件路径|
|file_type|varchar(16)|not null|pdf/docx/txt/md|
|file_size|bigint||文件大小|
|checksum|varchar(128)||文件摘要，用于去重|
|parent_document_id|uuid|fk documents(id)|父文档，用于版本链|
|document_version|varchar(32)|not null default '1.0'|文档版本|
|is_latest|boolean|not null default true|是否最新版本|
|parse_status|varchar(16)|not null default 'pending'|解析状态|
|parse_error|text||解析失败信息|
|parse_retry_count|int|not null default 0|解析重试次数|
|parse_strategy_version|varchar(32)||解析策略版本|
|ingested_by|uuid|fk users(id)|上传人|
|deleted_at|timestamptz||软删除时间|
|created_at|timestamptz|not null|创建时间|
|updated_at|timestamptz|not null|更新时间|

### 索引建议

- `idx_documents_doc_type`
- `idx_documents_company_id`
- `idx_documents_company_code`
- `idx_documents_publish_date`
- `idx_documents_parse_status`
- `idx_documents_source`
- `idx_documents_parent_document_id`
- `idx_documents_is_latest`
- 组合索引：`(company_code, doc_type, publish_date desc)`
- 组合索引：`(doc_type, parse_status)`
- trigram 索引：`title`

### 设计原因

- `source_id` 是引用系统和溯源系统的基础
- `company_id` 用于接主数据，`company_code/company_name` 作为快照保留
- `checksum + source_id` 组合唯一更适合去重
- `parent_document_id/document_version/is_latest` 为文档版本预留
- `parse_retry_count` 方便后续自动重试

## 5.4 `document_chunks`

### 用途

- 存储文档切分后的片段
- 存储检索和 RAG 直接使用的数据
- 存储结构化抽取结果

### 建议字段

|字段|类型|约束|说明|
|---|---|---|---|
|id|uuid|pk|chunk 主键|
|document_id|uuid|fk documents(id) not null|所属文档|
|chunk_id|varchar(128)|not null|片段编号|
|chunk_index|int|not null|片段顺序|
|chunk_text|text|not null|片段正文|
|content_type|varchar(32)|not null default 'paragraph'|内容类型|
|page_no|int||页码|
|position_start|int||原文起始位置|
|position_end|int||原文结束位置|
|char_start|int||字符起始位置|
|char_end|int||字符结束位置|
|section_title|varchar(256)||所属章节标题|
|section_path|text||章节路径，如“第三章>经营分析>盈利能力”|
|token_count|int||估算 token 数|
|summary_text|text||片段摘要|
|keyword_text|text||关键词检索辅助字段|
|keywords_json|jsonb||关键词抽取结果|
|entities_json|jsonb||实体抽取结果|
|metadata_json|jsonb||其他结构化元数据|
|is_important|boolean|not null default false|是否重点片段|
|embedding|vector(1024)||向量，维度按模型调整|
|created_at|timestamptz|not null|创建时间|

### 索引建议

- `idx_document_chunks_document_id`
- `idx_document_chunks_chunk_index`
- `idx_document_chunks_page_no`
- `idx_document_chunks_content_type`
- `idx_document_chunks_is_important`
- 组合索引：`(document_id, page_no, chunk_index)`
- GIN 索引：`entities_json`
- trigram 索引：`keyword_text`
- 向量索引：`ivfflat` 或后续合适索引

### 设计原因

- `chunk_index` 用于还原上下文顺序
- `page_no` 和位置字段用于引用定位
- `content_type/section_path/entities_json` 用于提升检索与摘要质量
- `metadata_json` 用于承接不同文档类型的额外解析结果

### 建议字段

## 5.7 `tasks`

### 用途

- 记录总控 Agent 接收到的任务
- 支撑任务路由、历史记录、问题排查

### 建议字段

|字段|类型|约束|说明|
|---|---|---|---|
|id|uuid|pk|任务主键|
|task_type|varchar(32)|not null|如 `qa`、`memo_generate`、`daily_report`、`document_ingest`|
|task_title|varchar(256)||任务标题|
|input_text|text||用户原始输入|
|input_payload|jsonb||结构化输入|
|output_payload|jsonb||最终结构化输出|
|result_summary|text||结果摘要|
|status|varchar(16)|not null default 'pending'|任务状态|
|priority|varchar(16)|not null default 'normal'|优先级|
|route_agent|varchar(64)||路由到的 Agent|
|callback_url|text||异步任务完成后的回调地址，第一期可先只存不调用|
|request_id|varchar(128)||请求链路 ID，对应 `X-Request-ID`|
|created_by|uuid|fk users(id)|发起人|
|started_at|timestamptz||开始时间|
|finished_at|timestamptz||结束时间|
|error_message|text||错误信息|
|created_at|timestamptz|not null|创建时间|
|updated_at|timestamptz|not null|更新时间|

### 索引建议

- `idx_tasks_task_type`
- `idx_tasks_status`
- `idx_tasks_route_agent`
- `idx_tasks_created_by`
- `idx_tasks_created_at`
- 组合索引：`(status, created_at desc)`
- `idx_tasks_request_id`

### 设计原因

- 一期虽不做复杂编排，但任务记录必须先有
- `output_payload/result_summary` 能减少排查和展示成本
- `/qa/ask` 是同步等待任务完成的便捷入口，底层也应写入 `tasks`
- `/tasks` 是异步统一入口，`callback_url` 为后续 Webhook 预留扩展位

## 5.8 `task_runs`

### 用途

- 记录任务在 Agent 或 Skill 层的执行过程
- 支撑调试、回放、问题定位

### 建议字段

|字段|类型|约束|说明|
|---|---|---|---|
|id|uuid|pk|运行记录主键|
|task_id|uuid|fk tasks(id) not null|所属任务|
|run_type|varchar(16)|not null|`agent` / `skill`|
|run_name|varchar(128)|not null|执行单元名称|
|input_payload|jsonb||输入参数|
|output_payload|jsonb||输出结果摘要|
|status|varchar(16)|not null|执行状态|
|duration_ms|int||耗时|
|error_message|text||错误信息|
|created_at|timestamptz|not null|创建时间|

### 设计原因

- 多 Agent 架构最容易出问题的地方就是“到底哪一步错了”
- 这张表能把总控 Agent、研究 Agent、报告 Agent 和 Skill 调用全部串起来

### 建议字段

|字段|类型|约束|说明|

|company_code|varchar(32)||公司代码快照|

|parent_asset_id|uuid|fk research_assets(id)|父资产|
|root_asset_id|uuid|fk research_assets(id)|根资产|
|version|int|not null default 1|版本号|
|status|varchar(16)|not null default 'draft'|状态|
|task_id|uuid|fk tasks(id)|来源任务|
|created_by|uuid|fk users(id)|创建人|
|updated_by|uuid|fk users(id)|更新人|
|deleted_at|timestamptz||软删除时间|
|created_at|timestamptz|not null|创建时间|
|updated_at|timestamptz|not null|更新时间|

### 索引建议

- `idx_research_assets_asset_type`
- `idx_research_assets_company_id`
- `idx_research_assets_company_code`
- `idx_research_assets_status`
- `idx_research_assets_created_at`
- `idx_research_assets_parent_asset_id`

## 5.16 `request_id` 与链路追踪

API v2 约定：

- 客户端可通过 Header `X-Request-ID` 传入请求 ID
- 如果客户端未传，服务端生成一个请求 ID
- 错误响应中的 `request_id` 与 `X-Request-ID` 保持一致
- 后端日志、`tasks.request_id`、必要的 `task_runs` 日志应记录同一个 ID

这样设计的原因：

- 本地开发排查问答、解析、Agent 调用链路时，能从接口响应一路追到日志和任务记录
- 后续接入前端或队列时，不需要重新设计链路追踪字段

## 7. 约束与校验建议

### 7.1 唯一约束

- `documents.source_id`
- `company_basic_info.company_code`
- `tags.tag_code`
- `prompt_templates.template_key`
- `system_configs.config_key`
- `asset_revisions(asset_id, version)`

说明：

- `documents.source_id` 是文档来源唯一标识，作为主去重约束。
- `documents.checksum` 用于检测重复上传，建议建普通索引，不建议与 `source_id` 再做组合唯一，避免两个去重语义重叠。

### 7.2 非空约束

- 文档标题、文档类型、文件路径
- chunk 正文
- 任务类型、任务状态
- 资产类型、资产标题、正文
- 标签编码、标签名称、标签类型
- 公司代码、公司名称

### 7.3 数据完整性

- `document_chunks.document_id` 必须存在
- `document_tags.document_id/tag_id` 必须存在
- `task_runs.task_id` 必须存在
- `citations.document_id` 必须存在
- `asset_sources.asset_id` 必须存在
- `research_asset_tags.asset_id/tag_id` 必须存在

### 7.4 外键删除策略

|关系|建议策略|原因|
|---|---|---|
|`documents -> document_chunks`|`ON DELETE CASCADE`|文档物理删除时 chunk 无独立意义|
|`documents -> document_tags`|`ON DELETE CASCADE`|文档删除后标签关系应清理|
|`tags -> document_tags`|`ON DELETE CASCADE`|标签删除后关系表应清理|
|`tags -> research_asset_tags`|`ON DELETE CASCADE`|标签删除后资产标签关系应清理|
|`tasks -> task_runs`|`ON DELETE CASCADE`|任务删除后运行明细同步清理|
|`tasks -> research_assets`|`ON DELETE SET NULL`|资产需要保留，来源任务可置空|
|`research_assets -> asset_revisions`|`ON DELETE CASCADE`|资产物理删除后版本无独立意义|
|`research_assets -> asset_sources`|`ON DELETE CASCADE`|资产来源关系随资产清理|
|`research_assets -> citations`|`ON DELETE CASCADE`|资产引用随资产清理|
|`documents -> citations`|`ON DELETE CASCADE`|文档物理删除后引用无法回溯|
|`document_chunks -> citations`|`ON DELETE SET NULL`|引用可保留文档和快照文本，但 chunk 指针置空|

补充：

- 一期业务删除默认软删除，即写入 `deleted_at`。
- 以上策略只在物理删除时生效。

## 8. 索引策略补充说明

### 8.1 索引优先级

P0 必建索引：

- `documents.source_id` 唯一索引
- `documents.company_code`
- `documents.doc_type`
- `documents.parse_status`
- `documents.publish_date`
- `document_chunks.document_id`
- `document_chunks(document_id, chunk_index)` 唯一索引
- `tasks.status`
- `tasks.created_at desc`
- `task_runs.task_id`
- `research_assets.company_code`
- `research_assets.asset_type`
- `research_assets.status`
- `citations.task_id`
- `citations.asset_id`
- `prompt_templates.template_key` 唯一索引
- `system_configs.config_key` 唯一索引

P2 有数据量后再加：

- `documents.title` trigram 索引
- `document_chunks.keyword_text` trigram 索引
- `document_chunks.entities_json` GIN 索引
- `company_basic_info.company_short_name` trigram 索引
- `tags.tag_type`、`tags.parent_id`
- `documents(company_code, doc_type, publish_date desc)`
- `research_assets(company_code, asset_type, created_at desc)`
- `tasks(status, created_at desc)`
- 向量索引 `ivfflat` 或后续 HNSW 索引

这样分级的原因：

- 初期数据量小，过多索引会拖慢写入和迁移。
- 先保证主流程查询可用，等样本文档和调用量上来后再补 P2 索引。

### 8.2 全文检索与模糊检索

建议第一期使用 `pg_trgm` 提供标题和关键词模糊搜索能力：

- `documents.title`
- `document_chunks.keyword_text`
- `company_basic_info.company_short_name`

### 8.3 组合索引设计原则

第一期常见查询路径：

- 公司 + 文档类型 + 日期
- 资产类型 + 公司 + 创建时间
- 任务状态 + 时间倒序
- 标签过滤 + 文档/资产列表

因此组合索引应优先围绕这些路径设计。

### 8.4 分区表预留

一期先不做分区，但应在设计上预留后续按 `created_at` 做月分区的对象：

- `tasks`
- `task_runs`
- `citations`
- `document_chunks`
