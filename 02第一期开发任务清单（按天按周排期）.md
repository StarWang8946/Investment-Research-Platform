# 第一期开发任务清单（按天按周排期）

> 适用阶段：第一期“研究信息中台 MVP”
> 
> 适用对象：1 人、本地开发
> 
> 目标：把第一期能力拆成可按天执行的开发任务，并说明关键技术为什么这样选

---

## 1. 排期说明

### 1.1 目标

第一期的真正目标不是“把所有多 Agent 架构都写完”，而是先跑通最小闭环：

- 文档接入
- 文档解析
- 检索问答
- 总控 Agent 路由
- 研究 Agent 输出
- 报告协作 Agent 输出
- 研究资产沉淀

### 1.2 节奏建议

- 总周期按 `6 周` 规划
- 每周 `5 个开发日`
- 每天只做 `1-2 个主任务`
- 每周末必须有可验证产物

### 1.3 一期最终交付标准

- 能上传和解析文档
- 能做带引用问答
- 能生成研究备忘录
- 能生成晨会日报
- 能保存研究资产
- 总控 Agent 能在“问答 / 备忘录 / 日报 / 入库”之间做基础路由

## 2. 为什么第一期采用这些技术

### 2.1 `FastAPI`

原因：

- 适合单人快速开发 API
- 异步支持较好，后续接长任务更自然
- 自动生成接口文档，调试成本低

不选重型框架的原因：

- 第一阶段重点是业务闭环，不是复杂企业级框架能力

### 2.2 `PostgreSQL`

原因：

- 可以同时承载文档元数据、任务记录、研究资产
- 后续二期三期还可以继续扩展，不会推倒重来
- 配合 `pgvector` 可以减少一开始引入独立向量数据库的复杂度

### 2.3 `pgvector`

原因：

- 向量和业务数据放在一个库里，单机开发最省事
- 方便做文档、chunk、引用、研究资产的统一查询
- 一期数据量不大，没必要先上独立向量服务

### 2.4 `Streamlit` 或 `Next.js`

建议：

- 如果目标是先验证业务闭环，用 `Streamlit`
- 如果目标是后续直接演进成产品界面，用 `Next.js`

原因：

- `Streamlit` 开发极快，适合单人原型
- `Next.js` 更接近正式产品，但开发成本更高

### 2.5 `pymupdf + python-docx + unstructured`

原因：

- 这几类库覆盖第一期主流文档类型
- PDF、DOCX、TXT/Markdown 都能处理
- 比自己从头做解析稳定得多

### 2.6 外部 LLM + 本地 RAG

原因：

- 第一阶段核心不是自建模型，而是“让模型基于本地知识工作”
- 外部模型推理能力强，接入快
- 本地 RAG 保证文档可信和引用可追踪

## 3. 6 周总排期总览

|周次|主题|核心产出|
|---|---|---|

## 4. 第 1 周：项目骨架与数据库

### 周目标

- 建立项目基础结构
- 跑通后端服务
- 连通数据库
- 完成一期核心表设计

### Day 1：项目初始化

任务：

1. 建项目目录结构
2. 初始化 Python 环境
3. 建 `app/`、`api/`、`services/`、`agents/`、`skills/`、`repositories/`、`models/`
4. 建 `.env.example`
5. 约定配置加载方式

为什么这么做：

- 一开始就做模块边界，后续增加 Agent/Skill 时不会混乱
- 配置和代码分离，后续切换模型、数据库更容易

当日产出：

- 项目可运行
- 目录结构固定

### Day 2：FastAPI 骨架

任务：

1. 搭建 FastAPI 主应用
2. 增加健康检查接口
3. 增加基础路由结构
4. 增加统一异常处理
5. 增加日志初始化

为什么用 FastAPI：

- API 风格天然适合 Agent、Skill、文档、任务这些模块化接口
- Swagger 文档自动生成，单人开发调试效率高

当日产出：

- `GET /health`
- 基础 API 服务可启动

### Day 3：数据库接入

任务：

1. 接 PostgreSQL
2. 建 SQLAlchemy 基础模型
3. 定义数据库会话管理
4. 加入迁移工具

为什么用 PostgreSQL：

- 第一阶段需要结构化保存文档元数据、引用、任务、研究资产
- 二期三期还能继续复用

当日产出：

- 服务可连接数据库
- 可创建基础表

### Day 4：一期核心表设计

任务：

1. 设计 `users`、`company_basic_info`、`documents`
2. 设计 `document_chunks`
3. 设计 `tags`、`document_tags`、`research_asset_tags`
4. 设计 `tasks`、`task_runs`
5. 设计 `research_assets`、`asset_revisions`、`asset_sources`
6. 设计 `citations`、`prompt_templates`、`system_configs`

为什么先做表：

- 一期多数功能都围绕文档、chunk、引用、任务展开
- 公司主数据、标签体系和研究资产版本如果不先定，后面返工成本很高
- 表结构不清楚，后续 Agent 和 Skill 容易返工

当日产出：

- 首版数据库 schema

### Day 5：本周收尾

任务：

1. 写数据库初始化脚本
2. 写最小 README
3. 跑通本地启动流程
4. 记录一期配置项

当周验收：

- 后端能启动
- 数据库能连上
- 核心表已经建好

## 5. 第 2 周：文档接入与解析

### 周目标

- 文档可上传
- 文档可保存
- 文档可解析
- 文档可切分并入库

### Day 6：文档上传接口

任务：

1. 实现 `POST /api/v1/documents`
2. 限制文件类型
3. 保存原始文件路径
4. 录入基础元数据
5. 兼容旧路径 `POST /documents/upload`

为什么先做上传：

- 没有原始文档，后面的解析、检索、问答都没有输入

当日产出：

- 可上传 PDF/DOCX/TXT/MD
- 上传后返回 `document_id`、`source_id`、`parse_status`

### Day 7：PDF/DOCX/TXT 解析

任务：

1. 接 `pymupdf`
2. 接 `python-docx`
3. 实现 TXT/MD 文本读取
4. 输出统一文本结构

为什么这样选：

- `pymupdf` 解析 PDF 稳定、速度快
- `python-docx` 适合处理 Word 文档
- 统一输出结构，后续 chunk 切分逻辑不用区分来源

当日产出：

- 各类文档都能提取文本

### Day 8：文本清洗与切分

任务：

1. 清理空白、分页噪声、页眉页脚干扰
2. 实现 chunk 切分
3. 保留页码和片段位置
4. 生成 `chunk_id`

为什么要自己控制切分：

- RAG 质量高度依赖 chunk 质量
- 金融文档里表格、段落、标题位置很重要

当日产出：

- 文档可切成稳定片段

### Day 9：文档与 chunk 入库

任务：

1. 保存文档记录
2. 保存 chunk 记录
3. 保存 `source_id`
4. 做去重或重复上传判断

为什么这一步要早做：

- 后续 embedding、检索、引用都依赖稳定的 chunk 主键

当日产出：

- 文档和片段已入库

### Day 10：文档处理流水线

任务：

1. 串起“上传 -> 解析 -> 切分 -> 入库”
2. 增加处理状态字段
3. 增加失败重试机制
4. 记录日志
5. 实现 `POST /api/v1/documents/{document_id}/ingest`

为什么要做流水线：

- 一期虽小，但文档处理至少是一个标准流程
- 提前流水线化，后面接 Agent 更顺

当周验收：

- 上传一份文档后可完成入库
- 失败文档可定位错误
- 可查询文档列表、文档详情、chunk 列表和预览文本

## 6. 第 3 周：检索与 RAG 问答

### 周目标

- 文档可生成 embedding
- 可做混合检索
- 可生成带引用的问答结果

### Day 11：embedding 接入

任务：

1. 选定 embedding 模型
2. 实现 embedding 生成接口
3. 将 embedding 写入数据库

为什么一期先做 embedding：

- 没有 embedding，就没有语义检索
- 研究问答不能只靠关键词匹配

当日产出：

- chunk 可以生成向量并写库

### Day 12：pgvector 检索

任务：

1. 启用 `pgvector`
2. 实现向量召回
3. 验证 top-k 结果

为什么用 `pgvector`：

- 与 PostgreSQL 一体化，单机维护成本最低
- 当前数据量不大，足够支撑一期

当日产出：

- 可按语义检索相似 chunk

### Day 13：关键词检索与混合召回

任务：

1. 实现标题/正文关键词检索
2. 融合向量召回结果
3. 增加简单排序策略
4. 检索接口只暴露 `query`、`filters`、`top_k` 等必要参数

为什么不用纯向量检索：

- 股票代码、会计科目、公司简称这类关键词很关键
- 混合召回对金融文本更稳

当日产出：

- 召回质量明显优于单一检索

### Day 14：RAG 问答链路

任务：

1. 实现 `POST /api/v1/qa/ask`
2. 检索相关 chunk
3. 拼接上下文
4. 调用 LLM 生成答案
5. 底层写入 `tasks`，同步接口等待任务完成后返回

为什么先做 RAG 再做 Agent：

- Agent 的核心价值是调度能力
- 如果底层问答链路还没稳定，先做 Agent 只会增加复杂度

当日产出：

- 给定问题可返回基于资料的答案

### Day 15：引用系统

任务：

1. 为答案生成引用编号
2. 记录 `source_id`、`chunk_id`
3. 输出引用列表
4. 落库引用记录
5. 明确 `answer` 中 `[1]`、`[2]` 与 `citations[].citation_no` 对齐

为什么一期必须做引用：

- 没有引用，这个平台只是聊天工具，不是投研工具

当周验收：

- 问答结果可返回引用
- 可回到原文片段
- 任务表能追踪问答执行状态

## 7. 第 4 周：总控 Agent 与研究 Agent

### 周目标

- 建立最小多 Agent 架构
- 实现总控 Agent
- 实现研究 Agent

### Day 16：Agent 抽象设计

任务：

1. 定义 Agent 基类
2. 定义 Skill 调用协议
3. 定义任务输入输出结构

为什么先定义抽象：

- 多 Agent 的关键不是数量，而是接口统一
- 一期只做 3 个 Agent，但后面会扩展

当日产出：

- Agent/Skill 的基础协议

### Day 17：总控 Agent 最小版

任务：

1. 识别任务类型
2. 识别输入对象
3. 输出任务路由决策

为什么一期只做最小总控：

- 当前任务类型少，没必要上复杂 Planner
- 先做可解释的路由，比过度智能更稳

当日产出：

- 总控 Agent 可在 4 类任务间路由

### Day 18：研究 Agent

任务：

1. 对接检索 Skill
2. 对接摘要 Skill
3. 对接问答链路
4. 生成研究结论

为什么研究 Agent 要先落地：

- 一期主要价值就是研究辅助
- 它是后续量化和投资环节的上游

当日产出：

- 研究 Agent 可以执行研究问答和摘要任务

### Day 19：研究备忘录生成

任务：

1. 定义备忘录模板
2. 从多个文档抽取关键信息
3. 输出结构化备忘录
4. 通过 `template_key` 选择 Prompt 模板

为什么先模板化：

- 自由生成可读性不稳定
- 模板化更适合业务验证和后续沉淀

当日产出：

- 可生成首版研究备忘录

### Day 20：多 Agent 串联

任务：

1. 让总控 Agent 调研究 Agent
2. 总控 Agent 汇总结果
3. 记录任务日志
4. 完成 `POST /api/v1/tasks`、`GET /api/v1/tasks/{task_id}`、`GET /api/v1/tasks/{task_id}/runs`

当周验收：

- 用户提一个研究问题时，系统能通过总控 Agent 路由并返回研究结果

### 周目标

- 实现报告协作 Agent

- 报告协作 Agent 骨架可用

### Day 22：晨会日报模板

任务：

1. 定义日报模板结构
2. 汇总市场/公告/新闻/研究观点

为什么先做日报：

- 晨会日报是最标准、最容易验证价值的组织输出

当日产出：

- 可生成晨会日报

### Day 23：研究资产保存

任务：

1. 设计研究资产保存结构
2. 保存备忘录正文

为什么要做资产沉淀：

当日产出：

- 备忘录和日报都能保存

### Day 24：导出能力

任务：

1. Markdown 导出

3. 文件命名规则
4. 实现 `POST /api/v1/assets/{asset_id}/export`

为什么导出很重要：

- 研究成果最终要进入沟通和汇报场景
- 导出能力能直接提升可用性

当日产出：

- 结果可导出

### Day 25：历史记录和查看页

任务：

1. 研究资产列表
2. 任务记录列表
3. 文档详情查看
4. 公司选择和标签过滤

当周验收：

- 备忘录和日报不仅能生成，还能查、能看、能导出

## 10. 每周固定动作

每周都建议做这 4 件事：

1. 周一确认本周目标
2. 周三做一次中间演示
3. 周五做一次回归验证
4. 周五更新问题清单和下周计划

原因：

- 单人开发最容易陷入“忙了很多但没有闭环”
- 固定节奏能保证每周都有业务产出

## 12. 一期完成后能支撑什么

完成这份排期后，第一期系统会具备：

- 最小多 Agent 形态
- 知识驱动的研究问答
- 模板化研究输出
- 可沉淀的研究资产
- 为第二期量化研究提供输入基础

它还不是真正完整的投研平台，但已经是总架构中可运行的第一段。
