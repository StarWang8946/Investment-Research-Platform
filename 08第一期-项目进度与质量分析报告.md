## 执行摘要

**整体完成度：约 55%**

项目已建立起完整的 API 骨架和数据库结构（15 张表 + 82 个索引 + 24 个外键全部到位），核心的文档管理、标签、任务、资产、公司主数据、Prompt 模板等 API 均已实现。但关键的核心能力——**向量检索（假实现）、LLM 接入（未配置）、Agent/Skill 层（未实现）**——尚未完成，导致系统无法形成完整闭环。

**关键发现：**
- **数据库状态完好**：PostgreSQL 18.4 运行中，15 表 + 82 索引 + 24 外键与设计完全一致，已有测试数据（14 文档、23 chunks、7 公司）
- **服务层混合状态**：10/11 个服务文件是真实实现，但 `embeddings.py` 是致命假实现（哈希模拟向量），`qa.py` 服务文件缺失
- **SQL 质量有缺陷**：`ingest_document` 无事务管理（删 chunks 后插 chunks 不在同一事务），无连接池、无重试机制
- **架构存在偏差**：设计是 Agent→Skill→Service→Repository 多层，实际是 Service 直写 SQL
- **无测试、无前端、无认证授权机制**
- **核心结论：QA 功能从未成功运行**（`citations` 表为 0 行，因嵌入是假的）

### 2.1 功能测试评估

#### 2.1.1 文档管理模块

**实现文件：** `app/services/documents.py`（266 行）

**功能点：**
- ✅ 文档上传：支持 PDF/DOCX/TXT/MD 等格式
- ✅ 文档解析：调用 text.py 读取文本，split_chunks 切分
- ✅ 向量化：调用 embeddings.py 生成嵌入（**但嵌入是假的**）
- ✅ 入库：将 chunks 存入 document_chunks 表
- ✅ 列表查询：支持多条件过滤、分页
- ✅ 详情查询：返回文档信息 + chunk 数量
- ✅ Chunks 查询：分页返回 chunks
- ✅ 预览：返回文档前 N 个字符
- ✅ 删除：软删除

**质量问题：**
1. ⚠️ **嵌入是假的**：embeddings.py 使用哈希算法生成向量，不是真实语义嵌入，导致向量检索无意义
2. ⚠️ **解析质量未知**：text.py 中的 PDF 解析使用 PyMuPDF，但未处理表格、图片等复杂元素
3. ⚠️ **切分策略简单**：split_chunks 按字符数切分，未考虑语义边界
4. ⚠️ **无异步处理**：ingest_document 是同步的，大文件会超时
5. 🔴 **无事务管理**：ingest_document 中删除旧 chunks 后插入新 chunks 不在同一事务，插入失败会导致数据不一致

```python
def _embed_local_hash(text: str, dim: int) -> list[float]:
    vector = [0.0] * dim
    tokens = TOKEN_RE.findall(text.lower())
    if not tokens:
        return vector
    for token in tokens:
        digest = hashlib.blake2b(token.encode("utf-8"), digest_size=16).digest()
        index = int.from_bytes(digest[:4], "big") % dim
        sign = 1.0 if digest[4] % 2 == 0 else -1.0
        vector[index] += sign
    norm = math.sqrt(sum(value * value for value in vector))
    if norm == 0:
        return vector
    return [value / norm for value in vector]
```

- 用 token 哈希值填充向量，**语义相似的文本不会产生相似的向量**
- 触发条件：`.env` 中 `EMBEDDING_BASE_URL` 和 `EMBEDDING_API_KEY` 均未配置时自动降级
- 当前系统正在使用此假实现
